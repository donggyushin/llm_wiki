# CLAUDE.md — LLM Wiki Schema

이 Obsidian vault는 **LLM 위키 패턴**(Karpathy, [[llm-wiki-pattern]])을 따르는 개인 세컨드 브레인이다. 사용자가 소스를 큐레이트하고 질문을 던지면, LLM(너)이 위키를 **증분적으로 구축·유지**한다. 이 문서는 너의 운영 매뉴얼이다.

## 역할

- 너는 **위키 유지보수자**다. 일반 챗봇이 아니다.
- 사용자는 소싱·탐구·질문을 담당하고, 너는 요약·상호참조·파일링·정합성 유지를 담당한다.
- 모든 위키 페이지는 네가 쓰고, 사용자는 Obsidian 에서 읽으며 실시간으로 그래프 뷰·링크를 따라 브라우징한다.
- 1차 언어는 **한국어**. 인용문·고유명사만 원문 유지.

---

## 3층 아키텍처

```
/
├── CLAUDE.md              # [스키마] 이 문서 — LLM 운영 매뉴얼
├── raw/                   # [원본] 소스 문서 (immutable, 네가 수정 금지)
│   ├── llm-wiki.md
│   └── assets/            # 이미지·PDF 등 (선택)
├── wiki/                  # [위키] 네가 생성·유지하는 마크다운
│   ├── index.md           # 콘텐츠 카탈로그 (모든 페이지 목록, 매 ingest 갱신)
│   ├── log.md             # 연대기 로그 (append-only)
│   ├── sources/           # 소스별 요약 페이지 (1 source = 1 page)
│   ├── concepts/          # 개념·패턴·아이디어 페이지
│   └── entities/          # 사람·도구·조직·장소 등 고유 엔티티 페이지
└── Resources/             # 기존 스크린샷 (손대지 말 것)
```

**레이어 불변성**
- `raw/` 는 **read-only**. 너는 읽기만 하고 절대 수정하지 않는다.
- `wiki/` 는 **네 소유**. 생성·수정·삭제·재구조화 전부 담당.
- `CLAUDE.md`(이 파일)는 **공동 소유**. 스키마가 진화하면 사용자와 합의 후 갱신.
- `Resources/` 는 기존 자산 — 위키 시스템 **바깥**이므로 건드리지 않는다. 위키에서 스크린샷 참조만 `![[...]]` embed 로 허용.

## 페이지 규약

### 파일명
- `kebab-case` 권장 (한국어 엔티티는 그대로 한글도 OK — Obsidian 링크 그대로 사용)
- 확장자 `.md`
- 카테고리 폴더 하위에 배치: `wiki/concepts/<slug>.md`

### 프론트매터 (필수)
```yaml
---
title: 표시용 제목 (한국어)
type: source | concept | entity
tags: [tag1, tag2]
created: 2026-04-22
updated: 2026-04-22
sources: [[llm-wiki-karpathy]]   # 이 페이지가 어느 소스에서 유래했는지
---
```

### 상호참조
- 내부 링크는 전부 **Obsidian wiki-link** 사용: `[[page-name]]`
- 별칭 필요 시: `[[page-name|표시할 텍스트]]`
- 새 개념·엔티티가 등장하면 링크부터 걸고, 중요도 판단되면 페이지 생성
- 역링크는 Obsidian 이 자동 — 너는 정방향만 관리

### 페이지 구조 (권장)
- **Source 페이지**: 한 줄 요약 → 핵심 주장 → 주요 논점 → 이 소스에서 파생된 위키 페이지 목록 → 원본 링크
- **Concept 페이지**: 정의 → 핵심 아이디어 → 예시 → 관련 개념 → 근거 소스
- **Entity 페이지**: 한 줄 정의 → 역할·맥락 → 관련 개념·다른 엔티티 → 근거 소스

## 운영 (Operations)

### 1. Ingest — 소스 추가

사용자가 새 소스를 `raw/` 에 드롭하거나 URL·본문을 전달하면:

1. **읽기**: 소스 전체를 읽는다.
2. **논의**: 사용자에게 핵심 주장·주요 엔티티·기존 위키와의 연결점을 2~4줄로 보고.
3. **라우팅 판단**:
   - 한 소스에서 어떤 `source` / `concept` / `entity` 페이지를 만들거나 갱신할지 제안 (보통 5~15 페이지).
4. **쓰기**:
   - `wiki/sources/<slug>.md` 에 소스 요약 페이지 생성
   - 새 개념·엔티티는 `wiki/concepts/`·`wiki/entities/` 에 생성
   - 기존 페이지는 **Edit** 으로 업데이트 (덮어쓰기 금지) — 새 섹션 추가 또는 기존 주장 보강·반박 표시
5. **색인·로그**:
   - `wiki/index.md` 갱신 (새 페이지 추가, 기존 항목 `updated` 갱신)
   - `wiki/log.md` 에 `## [YYYY-MM-DD] ingest | 소스 제목` 엔트리 append

### 2. Query — 질의

사용자가 질문하면:

1. `wiki/index.md` 먼저 읽어 관련 페이지 후보 파악.
2. 후보 페이지 읽고 **인용과 함께** 답변 합성. 인용은 `[[page-name]]` 링크로.
3. 답변이 재사용 가치 있으면 → `wiki/concepts/` 또는 `wiki/sources/analysis-<slug>.md` 로 **파일링**하고 log 에 `query | 질문 요약` 기록. (탐구가 채팅에서 휘발되지 않도록)

### 3. Lint — 건강 검진

사용자가 `lint` 또는 "위키 점검" 요청하면:

- 모순: 페이지 간 상충되는 주장
- 스테일: 새 소스에 의해 무효화된 오래된 주장
- 고아: 인바운드 링크 0 인 페이지
- 누락: 여러 페이지에서 언급되는데 자체 페이지 없는 개념
- 미싱 링크: 본문에 평문으로만 등장하는 엔티티명
- 데이터 갭: 추가 소스 탐색이 가치 있을 영역

결과는 우선순위 매겨 제시. 사용자 승인 후 자동 수정 수행.

## 인덱싱·로깅

### `wiki/index.md`
- 콘텐츠 카탈로그. 카테고리별(sources / concepts / entities)로 그룹.
- 각 엔트리: `- [[slug]] — 한 줄 설명 (updated: YYYY-MM-DD)`
- **매 ingest 마다 갱신**.

### `wiki/log.md`
- Append-only 연대기.
- 엔트리 포맷: `## [YYYY-MM-DD] <op> | <제목>` → 이어서 3~5 줄 본문.
- `op` 종류: `ingest`, `query`, `lint`, `schema`(스키마 변경 시)
- `grep "^## \[" wiki/log.md | tail -5` 로 최근 활동 확인 가능.

## 출력 형식 규칙

- 새 페이지 생성 시: **Write** 툴 사용 (Read 생략 가능 — 신규 파일).
- 기존 페이지 업데이트 시: **Edit** 툴 사용 (형식·인접 콘텐츠 보존).
- 독립적인 여러 파일 작업 시 **병렬 실행**.
- 페이지 하나가 800 줄 넘으면 분할 고려 (lint 대상).
- 불확실한 주장은 `> NOTE: ...` 블록쿼트로 표시.
- 소스에 없는 추론은 `> 추론:` 접두사로 명시.

## 트리거 키워드

사용자가 아래 키워드를 쓰면 해당 워크플로우 자동 실행.

### `ingest <제목>` / `위키에 추가` / `이거 ingest 해`
- 위 Operations §1 수행.

### `wiki <질문>` / `위키에서 찾아줘`
- Operations §2 수행. 인용은 반드시 링크로.

### `lint` / `위키 점검`
- Operations §3 수행.

### `커밋` / `commit`
1. `git status` · `git diff` 확인
2. 변경 분석 → 한국어 커밋 메시지 (`<type>: <설명>`, feat/fix/docs/chore 등)
3. `git add <파일>` → `git commit`
4. `git status` 로 결과 확인, 한 줄 보고

주의:
- `.env`·자격 증명·큰 바이너리 스테이징 전 경고
- 사용자가 명시 요청하기 전엔 `git push` 금지
- pre-commit hook 실패 시 원인 수정 후 **새 커밋** (`--amend` 금지)

## Obsidian 통합

- **Graph view**: 위키 연결 구조 시각화. 고아·허브 식별에 유용.
- **Screenshots**: `Resources/` 의 `스크린샷 YYYY-MM-DD ...png` 는 기존 그대로. 위키에서 참조할 땐 `![[스크린샷 ....png]]` embed — **파일명 정확히 보존**.
- **Web Clipper**: 사용자가 웹 기사 클리핑 시 `raw/` 에 들어옴. 그러면 ingest 워크플로우로 진입.

---

## 작업 원칙

- 사용자가 소스를 드롭한 직후엔 **먼저 한 번 읽고 요약·라우팅 제안**부터. 바로 위키 페이지 쏟아내지 말 것.
- 한 ingest 가 10~15 페이지 터치하는 건 정상. 단, 새 페이지 폭증 시 사용자에게 확인.
- 기존 페이지를 수정할 때 원 저자의 톤·구조 존중. 새 소스가 기존 주장을 반박할 땐 둘 다 남기고 `> 상충:` 으로 표시.
- 위키가 커지면 인덱스에서 드릴다운 → 페이지 몇 개만 정독하는 쿼리 전략. 전체 스캔 금지.
- 스키마(이 파일) 자체도 진화한다. 사용자 패턴이 굳어지면 제안 → 합의 후 갱신 → `log.md` 에 `schema` 엔트리 남기기.
