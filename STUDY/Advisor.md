opus 는 좋지만 너무 비쌈, sonnet 은 비용은 저렴하지만 성능이 조금 아쉬움. 이걸 선택해야함 

## Adivisor 
작은 모델이 전체작업을 실행하다가, 정말 어려운 판단이 필요한 순간에만 큰 모델에게 물어본다. 
 ![[스크린샷 2026-04-21 오후 2.46.42.png]]
![[스크린샷 2026-04-21 오후 2.47.50.png]]
Executor가 스스로 해결하기 어려운 문제를 맞닥뜨렸을때 Advisor 에게 조언을 구함. Advisor 는 도구사용도 하지 않고 사용자에게 나오는 출력물도 생성하지 않음. 오직 Executor 에게 조언만 줌. 이 모든 과정이 하나의 API 로 통합. 복잡한 오케스트레이션 로직이 필요하지 않음. 

## 그럼 서브에이전트랑 뭐가 다른거지?
### Advisor vs sub agent
기존의 방식은 큰 모델이 오케스트레이터가 되어서 작업을 쪼갬. 근데 Advisor 는 하위 작업자들이 먼저 작업을 하고 상위 고급 모델한테 질문을 하는 방식.
Opus 만 사용하는건 비효율적임. 누적되는 컨텍스트 안에서 실제로 고차원적인 판단이 필요한 순간은 사실 그렇게 많지 않음. 전체의 10~20% 정도일뿐. 나머지는 단순하고 기계적인 실행일 뿐. 기존에는 이 모든 작업을 Opus 요금이 계속해서 쌓이게 됨.

![[스크린샷 2026-04-21 오후 2.55.27.png]]
위 이미지를 보면 알 수 있겠지만, 저비용 고효율임. 소넷만 쓰는 것 보다 Advisor 로 Opus 와 함께 사용하는 것이 더 저렴하면서도 높은 점수를 얻었음. 

## 사용방법
클로드 코드에서
```
/advisor -> opus 
```

### 프롬프트 권장사항
실질적인 작업을 시작하기 전에 먼저 advisor 부르기
작업이 끝났을때 한번더 advisor 부르기 
출력 토큰 줄이는 팁: advisor 는 100단어 이내로, 열거된 단계의 형태로만 응답하라. 

## AI 에이전트 아키텍쳐 설계 공식이 바뀐다 
누가 더 큰 모델을 만드느냐가 제일 중요했었음. 물론 아직도 중요함. 
하지만 이번에 앤트로픽이 던진 메시지는
"정답이 더 큰 모델이 아니라 더 똑똑한 시스템 아키텍쳐가 아닐까?"
이게 앞으로의 아키텍쳐 기본 공식이 될 수도 있다. 계층형으로 다양한 모델을 조합한다.
