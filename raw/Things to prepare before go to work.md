**AI 코딩 환경에 대한 너만의 입장 정리** (가장 임팩트 큰 준비)

리드가 "AI 관련 전문 지식과 노하우를 팀 내에서 공유해달라"고 명시했어. 첫 한 달 안에 이 카드를 한 번은 꺼내야 한다는 뜻. 출근 전에 이런 질문에 답할 수 있게 정리해두면 좋아:

- 내가 가장 잘 쓰는 AI 코딩 도구는? (Cursor / Claude Code / Copilot / Windsurf 등) 각각 언제 쓰고 언제 안 쓰나?
	- 작업할때 터미널에서 claude code 사용
	- 문서 정리할때 obsidian + claude code 조합
- AI가 잘 돌도록 코드베이스를 어떻게 구조화해봤나? (CLAUDE.md, .cursorrules, 모듈 경계, 테스트 자동 실행 등)
	- 모듈별로 CLAUDE.md 구조화. root 의 CLAUDE.md 는 최대한 간소하게. 그 외에는 다른 문서들을 참조하는 용도로 사용
	- markdown writer agent 를 두어서 code 변경시마다 문서 동기화를 의무적으로 시킴. 강제화하기 위해서 hook 사용
	- 테스트코드도 hook 을 걸어서 강제적으로 작성. TDD 기반 작업
	- docs 폴더 내부에 ADR 파일들을 정리
	- 컨텍스트 최적화를 위해서 무거운 mcp 사용 자제
		- 쉽게 구축 가능한 mcp 라면 직접 생성 혹은 script 파일로 대체 
- AI로 비즈니스 임팩트를 낸 구체 사례 1~2개 — 어떤 작업에 얼마나 시간을 줄였고, 무엇을 학습했나
	- slack message, figma, jira 티켓 url 기반으로 작업 티켓을 자동으로 생성해주는 스킬 활용
	- claude squad(tmux + worktree)로 병렬로 처리할 수 있는 작업들 병렬화 
- "AI에게 시키지 말아야 할 일"의 내 기준은? (이걸 가진 사람이 시니어로 보임)
	- **기준: 컨텍스트 깊이 — 암묵지 많은 작업은 AI에게 시키지 않는다.**
	- 코드만 봐서는 알 수 없는 정보가 결정에 필요한 영역에서 AI는 약하다. 구체적으로 세 가지:
		- **레거시 수정** — "왜 이렇게 짰는지"의 역사를 모르면, *의도된 이상함*을 버그로 오인하고 멀쩡한 코드를 "개선"한다. 그 이상함이 6개월 전 incident의 패치였을 수 있다.
		- **조직 정치가 얽힌 코드** — 팀 경계, 소유권, 책임 분담은 코드에 안 적혀 있다. AI가 "이건 여기로 옮기는 게 깔끔" 이라고 해도, 그 모듈은 다른 팀 소유라 PR이 막힌다.
		- **Production incident 근본원인 분석** — 로그 + 메트릭 + 사람의 기억 + 최근 배포 히스토리를 합쳐야 보인다. AI는 코드만 본다. "이게 원인일 것 같다" 라는 그럴듯한 답에 끌려가면 진짜 원인을 놓친다.
	- **그래서 AI 활용 방식**: 가설 brainstorm 까지만. *결정은 사람이 한다.*
		- 레거시 → AI에게 "이 코드의 가능한 의도 5가지" 까지만. 수정 여부는 git blame + 동료 확인 후.
		- 조직 정치 → AI에게 "기술적으로 깔끔한 안" 받고, 정치적 실현 가능성은 사람이 판단.
		- Incident → AI에게 코드 가설 받고, 메트릭/로그/배포 히스토리는 사람이 검증.
