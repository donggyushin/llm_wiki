# 직장 글쓰기 자주 틀리는 표현 30

## 핵심 요약

본인이 정리한 한국어 직장 글쓰기 안티패턴 30선. 6개 카테고리로 묶임:
이중수동·번역체 · 쿠션어·군더더기 · 모호한 시간·우선순위 · 자주 틀리는 맞춤법 · 잘못된 존칭·과한 격식 · 약한 요청·사과.

관통 원칙은 **명확성 = 친절**. 모호한 글은 예의처럼 보이지만 실제로는 상대에게 부담을 떠넘긴다.
"혹시 시간 되시면", "가능하시다면", "추후 공유드리겠습니다" 류는 **요청자의 결정 부담을 수신자에게 전가**한다.
좋은 직장 글쓰기는 **말투를 부드럽게** 만드는 것이 아니라 **상대가 다음 액션을 즉시 잡을 수 있게** 만드는 것이다.

> [!note] 본 위키 첫 비-기술 도메인 페이지
> 기존 위키는 LLM/Claude Code/AI 코딩에 집중. 이 소스는 **한국어 비즈니스 커뮤니케이션** 도메인의 첫 진입점.
> 추후 *plain Korean*, *기술 글쓰기*, *PR/이메일 템플릿* 등으로 확장 가능.

## A. 이중 수동·번역체 (1~5)

| 잘못 | 바로잡기 | 메모 |
|------|---------|------|
| 되어집니다 | 됩니다 | "되다"가 이미 수동 |
| 보여집니다 | 보입니다 | 같은 원리 |
| 사료됩니다 | 생각됩니다 / 판단됩니다 | "사료(思料)"는 일제 잔재 |
| ~에 있어서 | ~에서 / ~에 / ~의 경우 | 일본어 「において」 번역체 |
| 회의를 가졌습니다 | 회의했습니다 | 영어 "have a meeting" 번역체 |

상세: [[concepts/double-passive-translationese]]

## B. 쿠션어·군더더기 (6~10)

| 잘못 | 바로잡기 |
|------|---------|
| 다름이 아니라, ~ | (삭제) 본론 즉시 |
| 혹시 시간 되시면 | (삭제) + 마감 명시 |
| 한 번 봐주세요 | 무엇을 / 어디를 / 어떻게 명시 |
| 일단 ~ | (90% 삭제 가능) |
| 좀 ~ | (거의 항상 삭제) |

상세: [[concepts/cushion-words-redundancy]]

## C. 모호한 시간·우선순위 (11~15)

| 잘못 | 바로잡기 |
|------|---------|
| 추후 공유드리겠습니다 | "11/8(금)까지" 식 날짜 박기 |
| 빠른 시일 내 | 날짜 박기 |
| 조만간 | 날짜 박기 |
| ASAP | "X일 X시 전까지" |
| 시간 되실 때 | 마감 명시 |

상세: [[concepts/vague-time-deadlines]]

## D. 자주 틀리는 맞춤법 (16~20)

| 항목 | 정답 | 함정 |
|------|------|------|
| 결재 vs 결제 | 결재(서명) / 결제(돈) | 한자 다름 — 메일에 잘못 쓰면 치명적 |
| 다르다 vs 틀리다 | 다르다(different) / 틀리다(wrong) | "의견이 틀려요" ❌ |
| 됬다 → 됐다 | "됐다" (되었다 줄임) | "됬다"는 한국어에 없음 |
| 할거에요 → 할 거예요 | 띄어쓰기 + "거예요" | "거에요"는 비문 |
| 어떻해 → 어떡해 | "어떡해" (어떻게 해 줄임) | "어떻해"는 없음 |

> 어휘 단위 항목이라 별도 concept 페이지로 추출하지 않음. 필요 시 `concepts/korean-spelling-traps` 로 분리 가능.

## E. 잘못된 존칭·과한 격식 (21~25)

| 잘못 | 바로잡기 | 분류 |
|------|---------|------|
| 커피 나오셨습니다 | 커피 나왔습니다 | 사물 존칭 |
| 말씀이 계시겠습니다 | 말씀이 있겠습니다 | "계시다"는 사람 존칭 |
| 수고하셨습니다 (윗사람에게) | 고생 많으셨습니다 / 감사합니다 | 결례 |
| 외람되지만 | 한 가지 의견 드리면 | 과한 격식 |
| 송구하오나 | 양해 부탁드립니다 / 죄송합니다만 | 사극 톤 |

상세: [[concepts/object-honorifics-overformality]]

## F. 약한 요청·사과 (26~30)

| 잘못 | 바로잡기 |
|------|---------|
| 검토 부탁드립니다 | 무엇을 / 어떤 관점에서 / 언제까지 |
| 가능하시다면 | 가능 여부 회신 부탁드립니다 |
| 죄송합니다만, ~ | 필요할 때만 |
| 잘 부탁드립니다 | 무엇을 어떻게 (또는 "감사합니다"로 대체) |
| 확인 부탁드립니다 | 무엇을 무슨 기준으로 |

상세: [[concepts/action-spec-three-elements]] — 요청 3요소 (대상 / 관점 / 마감)

## 위키 내 다른 페이지와의 연결

이 페이지는 **위키 내 첫 비-기술 도메인**이지만, 메타 원칙에서 기존 페이지와 공명한다.

- [[concepts/code-is-not-cheap]] — Pocock의 *AI 시대 나쁜 코드는 가장 비싸다* 와 같은 결.
  나쁜 한국어 글쓰기도 가장 비싸다(상대의 시간을 갉아먹기 때문).
- [[concepts/ubiquitous-language]] — DDD 어휘 정렬 = 직장 글쓰기의 *모호 어휘 제거*. 같은 정신.
- [[concepts/fresh-context-principle]] — *비대한 컨텍스트 < 신선한 컨텍스트* = *긴 인사말 < 액션 명시*. 군더더기 삭제 = 명확성 가산.

> [!note] 도메인 교차 메모
> AI 코딩 위키와 한국어 글쓰기 노트가 같은 결로 정렬된다는 발견. *모호함을 제거하는 일* 은 기술 / 인간 양쪽 모두에서 핵심.
