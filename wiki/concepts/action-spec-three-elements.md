# 요청 3요소 — 무엇 / 관점 / 마감

## 핵심 요약

직장 글쓰기에서 가장 많이 망가지는 단위: **요청문**. "검토 부탁드립니다 / 확인 부탁드립니다 / 잘 부탁드립니다" 류는 표면상 정중해 보이지만, 받는 쪽은 *무엇을, 어떻게, 언제까지* 해야 할지 모른다.

처방: **모든 요청문에 3요소**.

1. **무엇을** (대상) — 어느 파일의 어느 부분
2. **어떤 관점에서** (검토 축) — 논리/문법/숫자 정합성/디자인 등
3. **언제까지** (마감) — 구체 날짜·시각

3요소가 다 들어가면 요청은 자동으로 친절해진다. 부드러운 어투는 부차.

---

## 5개 대표 항목 ([[sources/business-writing-30-mistakes]] 항목 26~30)

### 26. 검토 부탁드립니다 → 무엇을 / 어떤 관점에서 / 언제까지
✅ "3장 결론부 논리 흐름 위주로, 11/8 18시까지 검토 부탁드립니다"
가장 흔한 약한 요청. 받는 쪽은 *문서 전체를 다 봐야 하나? 어떤 깊이로? 언제까지?* 모두 모름.

### 27. 가능하시다면 → 가능 여부 회신 부탁드립니다
"가능하시다면" 은 거절 여지를 너무 열어둠. 회신 의무를 넣어야 일이 굴러감.
✅ "가능 여부 11/4(월) 12시까지 회신 부탁드립니다"

### 28. 죄송합니다만, ~ → 필요할 때만
습관처럼 붙이면 *또 사과네* 가 됨. 진짜 잘못한 경우만.
사과의 인플레이션이 일어나면, 정작 진짜 사과해야 할 때 무게가 사라진다.

### 29. 잘 부탁드립니다 → 무엇을 어떻게 (또는 "감사합니다"로 대체)
✅ "금요일 회의 자료 준비 잘 부탁드립니다" 정도로 구체화.
마무리 인사로만 쓸 거면 **"감사합니다"** 가 더 깔끔.

### 30. 확인 부탁드립니다 → 무엇을 무슨 기준으로
✅ "첨부 파일 5번 항목, 데이터 출처가 맞는지 확인 부탁드립니다"
"확인" 도 "검토" 와 같은 약한 요청 동사. 단독 사용 시 항상 3요소 보강.

## 요청 3요소 템플릿

```
[무엇] [어떤 관점에서] [언제까지] [동사: 검토/확인/회신] 부탁드립니다.
```

### 예시 변환

| Bad | Good |
|-----|------|
| "검토 부탁드립니다" | "3장 결론부, 논리 흐름 위주로, 11/8 18시까지 검토 부탁드립니다" |
| "확인 부탁드립니다" | "첨부 5번 항목, 데이터 출처 정합성 기준, 11/8 EOD 까지 확인 부탁드립니다" |
| "한 번 봐주세요" | "기획서 페이지 7~9, 사용자 시나리오 일관성 위주로, 내일 오전까지 봐주실 수 있을까요" |
| "회신 부탁드립니다" | "회의 가능 시간 3개 후보, 11/4(월) 12시까지 회신 부탁드립니다" |
| "잘 부탁드립니다" (마무리) | "감사합니다" |

## 왜 약한 요청이 결국 무례한가

표면: 부드러움 / 정중함.
실제: **결정 부담을 수신자에게 전가**.

받는 쪽 입장에서 약한 요청을 받으면 다음 작업을 강요당한다:
1. *어디까지 봐야 하지?* — 범위 추정
2. *어떤 깊이로 봐야 하지?* — 관점 추정
3. *언제까지 답해야 하지?* — 마감 추정
4. *답 안 해도 되나?* — 의무 추정

→ 이 모든 추정 비용은 발신자가 명세 한 줄로 없앨 수 있는 비용. 안 없앤 건 **친절의 부재**.

→ [[concepts/business-writing-clarity]] 의 *명확성=친절* 핵심 정리.

## 위키 내 메타 공명

이 개념은 위키 내 기술 페이지와 1:1 대응한다.

- [[concepts/plan-mode]] — *맞아? 묻지 말고 명시적으로 지시*. 요청문 = 인간 버전 plan.
- [[concepts/claude-md]] — AI 에 대한 *무엇을/어떤 기준으로/어떤 형식으로* 지시. 인간 요청과 같은 구조.
- [[concepts/harness-engineering]] — System Enforcement: 명세가 약하면 출력도 약함. 인간 협업도 동형.
- [[concepts/specs-to-code-antipattern]] — 명세만 고치고 코드 무시 = 요청만 던지고 액션 안 잡음. 같은 결.
- [[concepts/agent-teams]] — *사전 MD 패키징 의무*. 팀 리더가 워커에 컨텍스트를 명시 패키징하는 행위가 곧 *요청 3요소* 의 AI 버전.

> [!note] 도메인 교차 핵심 발견
> AI 에 좋은 프롬프트의 구조(*무엇/관점/형식/제약*) = 사람에게 좋은 요청문 구조(*무엇/관점/마감*).
> *프롬프트 엔지니어링과 한국어 직장 글쓰기는 같은 기술의 두 표면*.
