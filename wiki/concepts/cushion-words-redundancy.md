# 쿠션어·군더더기

## 핵심 요약

직장 글쓰기에서 **삭제만 해도 글이 좋아지는** 표현 묶음. "다름이 아니라 / 혹시 시간 되시면 / 한 번 / 일단 / 좀" 등.
표면상 부드러움을 더하는 것처럼 보이지만, 실제로는 **본론을 늦추거나 책임을 흐리는** 군더더기다.

핵심 처방: **거의 항상 삭제**. 의미는 그대로, 톤만 단단해진다.
부드러움이 필요한 자리는 쿠션어가 아니라 **구체적 마감/관점/대상** 으로 채워야 한다.

---

## 5개 대표 항목 ([[sources/business-writing-30-mistakes]] 항목 6~10)

### 6. 다름이 아니라 ~ → (삭제하고 바로 본론)
"다름이 아니라 부탁이 있어서요" → "A 건 부탁드립니다".
본론을 의도적으로 늦추는 마법의 군더더기. 받는 쪽 입장에선 *본론까지 한 박자 더 기다려야 함*.

### 7. 혹시 시간 되시면 → (삭제) + 마감 명시
"혹시 시간 되시면 봐주세요" → "금요일 오후 5시까지 검토 부탁드립니다".
거절 여지를 너무 열어두는 표현. 동시에 마감이 없으니 영원히 미뤄짐.
→ [[concepts/vague-time-deadlines]] 와 동시 위반.

### 8. 한 번 봐주세요 → 무엇을, 어디를, 어떻게
"한 번"은 부탁의 그림자. **무엇을 / 어디를 / 어떻게** 가 빠진 자리를 채우는 헛말.
→ "3페이지 결론 부분 논리 흐름 위주로 봐주세요"
→ [[concepts/action-spec-three-elements]] 와 직결.

### 9. 일단 → (90% 삭제 가능)
"일단 공유드립니다" → "공유드립니다". 의미 동일.
"일단" 은 *나중에 더 정제해서 보내겠다* 는 무의식적 변명을 깔지만, 받는 쪽에는 그 변명이 보이지 않는다.

### 10. 좀 → (거의 항상 삭제)
"좀 부탁드립니다", "좀 늦을 것 같아요". 빼도 의미 그대로, 톤만 단단해짐.
"좀" 은 자기 부탁의 무게를 스스로 깎으려는 본능. 깎을 필요가 없는 글이 좋은 글.

## 삭제 알고리즘

다음 표현 발견 시 우선 삭제 시도:

| 군더더기 | 삭제 후 영향 |
|---------|-------------|
| 다름이 아니라 | 본론이 즉시 시작 |
| 혹시 ~면 | 마감으로 치환 필요 |
| 한 번 | 액션 명세로 치환 필요 |
| 일단 | 무의미 — 그냥 삭제 |
| 좀 | 무의미 — 그냥 삭제 |
| 아무튼 | 무의미 — 그냥 삭제 |
| 그래도 | 거의 무의미 — 검토 후 삭제 |

→ 삭제만 해도 글이 90% 이상 같은 의미로 살아남으면 **그게 군더더기 증명**.

---

## 부드러움 ≠ 쿠션어

쿠션어를 빼면 차갑게 들릴까봐 걱정될 수 있다. 그러나 **부드러움의 진짜 출처**는 따로 있다:

1. **구체성** — "11/8 18시까지" 가 "혹시 시간 되시면" 보다 친절하다.
2. **이유 명시** — "X 미팅 자료 준비 차원에서" 가 "그냥 한 번" 보다 친절하다.
3. **감사 인사** — 마무리에 단정한 "감사합니다" 한 줄.

→ [[concepts/business-writing-clarity]] 의 *명확성 = 친절* 따름정리 3 *"삭제 = 톤 강화"* 정확히 이 메커니즘.

## 위키 내 메타 공명

- [[concepts/fresh-context-principle]] — 비대한 컨텍스트 < 신선한 컨텍스트. 글쓰기에선 *비대한 인사말 < 신선한 본론*.
- [[concepts/code-is-not-cheap]] — 의미 없이 늘어난 글자는 비용. 코드와 같은 결.
- [[concepts/rewind-pattern]] — Esc×2 로 실패 시도 도려내기 = 글쓰기에서 *쿠션어 도려내기* 의 정신적 쌍둥이.
