# What NOT to Delegate to AI

## 핵심 요약

**AI 에게 시키지 말아야 할 일을 가진 사람이 시니어로 보인다.** 단일 기준은 **컨텍스트 깊이** — 코드만 봐서는 알 수 없는 정보 (역사·조직 맥락·운영 데이터) 가 결정에 필요한 영역에서 AI 는 구조적으로 약하다. 그럴듯한 답을 내지만, 그 답이 *옳다고 검증할 근거* 가 코드 안에 없다. 3종 영역: **레거시 수정 · 조직 정치 코드 · Production incident 분석**. AI 활용은 *가설 brainstorm* 까지만, 결정은 사람.

> [!note] 시니어 시그널인 이유
> 주니어는 "AI 가 이것도 할 수 있다" 의 리스트를 자랑한다. 시니어는 "AI 에게 시키면 안 되는 영역" 의 리스트를 가진다. 후자는 **실패 모드의 직관** 이 있어야만 만들 수 있다 — 즉 본인이 AI 의 그럴듯한 거짓말에 한 번은 데인 흔적이다.

## 단일 기준 — 컨텍스트 깊이

> AI 는 **주어진 컨텍스트** 안에서만 추론한다.
> 결정에 필요한 정보가 *코드 바깥* (사람의 기억, 운영 메트릭, 조직의 비공식 합의) 에 있으면, AI 는 그 부재를 **인식하지 못한 채 답을 만든다**.

이게 핵심. *그럴듯한 답* 자체가 위험 신호 — 검증 채널이 코드 안에 없다.

## 3대 위임 금지 영역

### 1) 레거시 수정

**실패 모드**: "왜 이렇게 짰는지" 의 역사를 모르면 **의도된 이상함을 버그로 오인** 하고 멀쩡한 코드를 "개선" 한다.

**구체 사례 패턴**:

```python
# AI 가 보면 "이건 불필요한 분기야" 라고 판단할 수 있는 코드
if user.created_at < EPOCH_FIX_DATE:
    return legacy_format(user)
return modern_format(user)
```

이 분기가 6개월 전 데이터 마이그레이션 사고의 패치였다면? AI 는 모른다. 코드만 본다.

**대응**:
- AI 한계선: *"이 코드의 가능한 의도 5가지"* 까지만 받음
- 사람 판단: `git blame` + `git log -S` + 동료 확인 후 수정 여부 결정
- 시그널: 5년 이상 된 모듈, 의미 불분명한 분기, "이상한 if" 가 있는 코드

### 2) 조직 정치 얽힌 코드

**실패 모드**: 팀 경계·소유권·책임 분담은 코드에 안 적혀 있다. AI 가 *기술적으로 깔끔한 안* 을 제시해도, 그 모듈은 다른 팀 소유라 PR 이 막힌다. *Conway's Law* 의 다른 얼굴.

**구체 사례 패턴**:
- "이 함수를 공통 패키지로 추출하면 깔끔" — 그 패키지는 플랫폼 팀 소유, 외부 PR 거부 정책
- "이 두 서비스를 합치면 효율적" — 한쪽 서비스가 본부 KPI 에 묶여 있음
- "이 DB 스키마를 정규화하자" — 다른 팀이 비정규화된 형태에 의존

**대응**:
- AI 한계선: *"기술적으로 깔끔한 안"* 까지만
- 사람 판단: 정치적 실현 가능성·소유권 매핑·롤아웃 비용은 사람 검증
- 시그널: 팀 간 경계, 소유권 모호, 변경이 *기술적으론 쉬운데 안 일어나는* 영역

### 3) Production Incident 근본원인 분석

**실패 모드**: 로그 + 메트릭 + 사람 기억 + 배포 히스토리를 *모두 합쳐야* 보이는 원인을 AI 에게 *코드만* 보여 주고 묻는다. AI 가 그럴듯한 가설을 내면 사람이 거기에 끌려가 진짜 원인을 놓친다.

**구체 사례 패턴**:
- 5초마다 spike → AI 는 코드에서 retry 로직을 의심 → 실제 원인은 외부 cron job
- 특정 사용자만 실패 → AI 는 input validation 의심 → 실제 원인은 그 사용자 region 의 CDN cache 오염
- DB connection 고갈 → AI 는 connection leak 의심 → 실제 원인은 며칠 전 max_connections 변경

**대응**:
- AI 한계선: *코드 가설* 까지만
- 사람 검증: 메트릭 / 로그 / 배포 히스토리 / 운영 채널 기록과 *반드시* 교차
- 시그널: 재현 불가, 시간 의존, 환경 의존, 멀티 컴포넌트 얽힘

## 4축 분류 (확장 프레임)

3종 영역을 만든 *근본 축* 4개. 면접 답변·팀 가이드라인 작성 시 이 4축으로 일반화 가능:

| 축 | 질문 | AI 위임 금지 신호 |
|---|---|---|
| **검증 가능성** | 결과를 내가 검증할 수 있나 | 모르는 도메인 / 처음 보는 라이브러리 / 프로파일링 없는 최적화 |
| **되돌리기 비용** | 잘못 가면 되돌리기 비용이 얼마인가 | 모듈 경계 · 데이터 스키마 · API contract · 추상화 이름 · 의존성 추가 |
| **컨텍스트 깊이** | 결정에 코드 바깥 정보가 필요한가 | 레거시 / 조직 정치 / Production incident |
| **책임 소재** | 사고 시 *AI 가 했다* 가 변명이 되는가 | 보안 (auth/암호화/권한) / 결제·정산 / 마이그레이션 / 삭제·리셋 |

> [!note] 본 페이지의 초점
> 이 페이지는 *컨텍스트 깊이* 축에 집중. 나머지 3축은 인접 개념으로 분리 가능 (TODO).

## 면접 / 팀 공유용 한 줄 답변

> "AI 는 **'문제 정의' 와 '결정'** 을 시키지 않는다. 코드는 시키지만 *왜 이 코드인가* 는 내가 정한다. AI 가 '왜' 까지 정하기 시작하면 코드베이스의 일관성이 무너진다."

또는 구체적으로:

> "되돌리기 비싼 결정 (스키마·모듈 경계·추상화 이름), 검증 루프 없는 작업 (모르는 도메인), 책임 소재가 명확해야 하는 영역 (보안·결제), 그리고 컨텍스트 깊이가 코드 바깥에 있는 작업 (레거시·조직·incident) — 이 넷은 AI 가 초안을 내도 처음부터 다시 짠다."

## 안티 페어 — 이 기준이 *없는* 사람의 행동

- AI 가 낸 답에 *왜* 를 묻지 않고 그대로 머지
- Production incident 가 터졌을 때 가장 먼저 AI 에게 물어보고 답을 받아들임
- 레거시의 *이상한 분기* 를 발견하면 즉시 "리팩토링" PR 올림
- 다른 팀 소유 모듈에 AI 추천대로 변경을 시도하고 거부당한 후에야 소유권 확인
- "이거 AI 가 짰어" 를 결함의 변명으로 사용

- **TDD hook 강제** — 검증 루프 없으면 AI 코드를 못 믿음 → [[concepts/hooks]]
- **markdown writer hook 강제** — 문서/코드 drift 가 AI 환각의 원인 → [[concepts/hooks]]
- **MCP 다이어트** — 컨텍스트 부풀리면 AI 신뢰도 하락 → [[concepts/context-engineering#3-mcp-다이어트]]
